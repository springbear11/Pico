## 一、系统定位

### 1.1 在整体系统中的位置

PicoATE 是 ATE 软件栈中的**任务调度与执行引擎**，位于 UI 层与业务逻辑层之间。

```
┌──────────────────────────────────────────────────┐
│                  MES / 工厂系统                    │
│                   （未来接入）                      │
├──────────────────────────────────────────────────┤
│              UI 层（Qt Widgets，规划中）             │
│         消费 ExecutionReport 只读 DTO              │
├──────────────────────────────────────────────────┤
│              ★ PicoATE 任务引擎（本工程）★           │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │ JSON编译器│ │ 调度引擎  │ │ 设备连接管理器    │  │
│  │ +Plan    │ │ Scheduler│ │ DeviceSession    │  │
│  │ Builder  │ │          │ │ Manager          │  │
│  └──────────┘ └──────────┘ └──────────────────┘  │
│  ┌──────────────────────────────────────────────┐ │
│  │         ModuleRegistry / NodeRunner          │ │
│  └──────────────────────────────────────────────┘ │
├──────────────────────────────────────────────────┤
│              Transport 抽象层                      │
│  QProcess │ Persistent │ DllBridge │ NativeHost   │
├──────────────────────────────────────────────────┤
│          业务模块（DLL / Python / EXE）             │
├──────────────────────────────────────────────────┤
│        设备层（DMM / CAN / PSU / Fixture）          │
└──────────────────────────────────────────────────┘
```

### 1.2 核心设计约束

| 约束 | 含义 |
|------|------|
| 协议逻辑不进调度器 | CAN/LIN/DMM 命令解析属于业务模块或 Transport Host |
| UI 只消费只读 DTO | 使用 ExecutionReport，不读运行时内部状态 |
| 调度器拥有编排权 | Retry/Cleanup/Loop/Barrier/Resource 集中管理 |
| 模块返回中性结果 | 返回 ModuleResult，不返回 NodeResult 或 UI 数据 |
| 配置声明式 | 流程/策略/资源/输入用 JSON 描述，不编译进代码 |
| Transport 可替换 | DLL/Python/QProcess 都放在 IModuleTransport 接口后面 |

## 二、任务引擎核心功能

### 2.1 JSON 流程编译

引擎接收声明式 JSON 测试序列，编译为不可变执行计划（ExecutionPlan）：

```json
{
  "groups": [
    { "kind": "setup",   "steps": [ ... ] },
    { "kind": "main",    "steps": [ ... ] },
    { "kind": "cleanup", "steps": [ ... ] }
  ]
}
```

**编译过程：** JSON → SequenceDef（编辑模型）→ PlanBuilder → ExecutionPlan（不可变执行图）。
编译阶段完成节点生成、依赖边、控制边、清理区（CleanupRegion）和循环区（LoopRegion）的构建。
同时提供类型校验、未知字段警告、重复 ID 检测等诊断信息。

### 2.2 调度执行

引擎负责控制测试流程的执行顺序和运行规则：

| 能力 | 说明 |
|------|------|
| **依赖编排** | 按照 Setup → Main → Cleanup 顺序执行，Main 完成或失败后进入 Cleanup |
| **失败处理** | 每个步骤可配置重试次数（Retry）；失败后可选择继续、跳过、执行清理、停止 |
| **错误策略** | 区分三种失败类型（Fail / Error / Timeout），每种可独立配置处理方式 |
| **停止保护** | 用户请求停止后，跳过未执行步骤，强制执行 Cleanup 安全收尾 |
| **循环控制** | 支持 For 循环（from/to/step），循环游标由调度层维护，不修改执行计划 |
| **变量替换** | 支持 `${PROJECT_DIR}` 等编译期变量和 `${var.sampleIndex}` 等运行期变量 |

### 2.3 运行报告

所有执行结果汇总为只读 DTO（ExecutionReport），包含：

- **UUT 层**：每个被测单元的通过/失败状态
- **Step 层**：每个步骤的执行状态、耗时、重试次数
- **Attempt 层**：每次重试的具体结果、错误码和错误消息
- **Measurement 层**：测量名称、数值、单位、上下限、判定结果

UI 和 CLI 均通过同一套 DTO 消费结果，不接触运行时可变状态。

## 三、业务模块接入

### 3.1 模块接口

业务模块通过 `IModule` 接口接入引擎，无需了解调度内部状态：

```
引擎 → ActionNodeHandler → ModuleRegistry → IModule::execute(funcName, context) → ModuleResult
```

业务模块接收 `ModuleExecutionContext`（包含当前 UUT、输入参数、变量快照），返回 `ModuleResult`（Passed/Failed/Error/Timeout + 输出 + 测量值）。引擎负责将 ModuleResult 转换为内部 NodeResult。

### 3.2 当前重点：DLL 插件加载

首选接入方式为 **NativeHost + DLL**：

```
PicoATE 主进程
    ↓ QProcessTransport（子进程隔离）
PicoATE.NativeHost.exe
    ↓ QLibrary 加载 DLL
VendorDriver.dll
    ↓ 调用 PicoATE_Execute C ABI
```

DLL 只需导出以下 C 接口即可接入：

```c
extern "C" __declspec(dllexport)
int PicoATE_Execute(const char* requestJsonUtf8, char* responseJsonUtf8, int responseBufferSize);
```

请求和响应均使用 JSON 格式，与模块协议一致。DLL 崩溃只影响 NativeHost 子进程，不会拖死主程序。

### 3.3 后续扩展：Python / EXE

通过 Transport 抽象层（IModuleTransport），相同协议可扩展到其他语言：

| 扩展方向 | 实现方式 | 状态 |
|----------|----------|------|
| DLL 模块 | NativeHost.exe + QLibrary | 当前重点 |
| Python 脚本 | QProcessTransport + `python script.py` | 后续支持 |
| 独立 EXE | QProcessTransport + stdin/stdout JSON | 后续支持 |
| 长驻进程 | PersistentQProcessTransport（跨步骤保持状态） | 后续支持 |

业务模块不需要知道底层走的是 DLL、Python 还是 EXE——引擎对它们一视同仁。

### 3.4 设备连接管理

引擎提供设备连接生命周期管理（DeviceSessionManager），支持：

- 逻辑设备 ID（如 DMM1、CAN1）与真实地址解耦
- 连接复用（同一设备多次访问不重复连接）
- 工站级生命周期（Station 级设备开机连一次，持续复用）
- 健康检查和断线重连

工站设备信息通过 Station Config JSON 独立配置，与测试流程 JSON 分离。

## 四、扩展能力

### 4.1 多 UUT 并行调度

引擎支持同时执行多个被测单元（UUT）。每个 UUT 维护独立的运行状态和变量空间，引擎以 Round-robin 方式轮转推进各 UUT 的测试步骤。

### 4.2 资源互斥仲裁

多 UUT 共享仪器（如 DMM、PSU）时，引擎自动协调资源占用：

- **Exclusive**：独占模式，同一时间只有一个 UUT 能用
- **SharedRead / SharedWrite**：共享读写
- **Counted**：计数资源（如并行通道数量上限）

资源需求在 JSON 中声明，引擎在运行时自动排队和分配。

### 4.3 批次同步（Barrier）

多 UUT 可配置同步点，全部到达后才继续执行。支持多种策略：

- **到达策略**：WaitAll（等齐）/ DropFailed（移出失败者）/ Quorum（达到比例即可）
- **释放策略**：Lockstep（同步释放）/ Latch（各自继续）/ Cohort（分组）
- **失败策略**：FailBarrier / RemoveFailedMember / ContinueWithWarning

## 五、工站配置

工站信息与测试流程分离，通过独立的 Station Config JSON 管理：

```json
{
  "stationId": "bench-01",
  "devices": [
    {
      "deviceId": "DMM1",
      "deviceType": "DMM",
      "driverId": "keysight.34465a.visa",
      "address": "USB0::0x0957::0x0607::MY59001234::INSTR",
      "lifetime": "Station"
    }
  ]
}
```

换工站或换仪器只需改此配置文件，测试流程 JSON 不受影响。

## 六、输入输出总览

**输入：**

| 输入 | 格式 | 提供方 |
|------|------|--------|
| Sequence JSON | JSON 文件，描述测试步骤和策略 | 测试工程师/项目团队 |
| Station Config JSON | JSON 文件，描述工站设备地址 | 工站配置 |
| UUT 数量 | 运行参数 | 操作员 |
| 业务 DLL | C ABI DLL | 项目团队 |

**输出：**

| 输出 | 格式 | 消费者 |
|------|------|--------|
| ExecutionReport | 结构化数据（UUT/Step/Attempt/Measurement） | UI / CLI |
| CLI 日志 | stdout 打印执行过程和结果 | 调试/日志 |
