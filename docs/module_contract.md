# PicoATE External Module Contract

This document defines the first external module contract for process-based
business logic modules.

The goal is simple: a project-specific module can run outside the PicoATE
scheduler process as long as it accepts one JSON request line on stdin and
writes one JSON response line to stdout.

## Process Contract

| Channel | Contract |
|---------|----------|
| stdin | One compact or normal JSON object per line. |
| stdout | One JSON response object per request line. |
| stderr | Diagnostic text only; PicoATE may include it in transport errors. |
| exit code 0 | Host completed normally. |
| non-zero exit code | Transport error. |
| timeout | Transport timeout; scheduler maps it to `ModuleOutcome::Timeout`. |

The current `QProcessTransport` starts a short-lived process for each module
call. This keeps the first implementation easy to reason about and isolates
host crashes from the scheduler process. A persistent process transport can be
added later behind the same `IModuleTransport` interface.

## Sequence Module Binding

A sequence can declare external process modules at the top level:

```json
{
  "moduleBindings": [
    {
      "moduleId": "external.echo",
      "transport": "qprocess",
      "program": "${SEQUENCE_DIR}/modules/echo_host.exe",
      "arguments": [],
      "timeoutMs": 3000
    }
  ]
}
```

Action steps then use the same `moduleId`:

```json
{
  "id": "external-echo",
  "kind": "action",
  "moduleId": "external.echo",
  "function": "echo",
  "inputs": {
    "value": "hello"
  }
}
```

The scheduler still sees only `IModule`. Runtime setup resolves the binding and
registers a `TransportModuleAdapter` backed by `QProcessTransport`.

## Request JSON

`ModuleTransportRequest` is serialized as:

```json
{
  "traceId": "external.echo:8f8d9f0a",
  "moduleId": "external.echo",
  "function": "echo",
  "context": {
    "uutId": "uut-1",
    "frameId": "root",
    "attemptId": "attempt-1",
    "attemptIndex": 0,
    "inputs": {
      "value": "hello"
    },
    "parameters": {
      "moduleId": "external.echo",
      "function": "echo"
    },
    "variables": {}
  }
}
```

| Field | Meaning |
|-------|---------|
| `traceId` | Correlation id generated by PicoATE for logging across processes. |
| `moduleId` | Logical module id from the sequence step, such as `project.can`. |
| `function` | Function/action name from the sequence step. |
| `context.uutId` | UUT currently executing this step. |
| `context.frameId` | Execution frame id. |
| `context.attemptId` | Attempt id generated by the scheduler. |
| `context.attemptIndex` | Zero-based attempt index. |
| `context.inputs` | Project-defined input payload from JSON/configuration. |
| `context.parameters` | Full action payload seen by the scheduler. |
| `context.variables` | Runtime variables snapshot, currently minimal. |

## Response JSON

`ModuleTransportResponse` is expected as:

```json
{
  "outcome": "Passed",
  "outputs": {
    "actualVoltage": 4.999
  },
  "measurements": {
    "name": "VOUT",
    "value": 4.999,
    "unit": "V",
    "lowerLimit": 4.9,
    "upperLimit": 5.1,
    "status": "Passed"
  },
  "errorCode": "",
  "errorMessage": ""
}
```

| Field | Meaning |
|-------|---------|
| `outcome` | `Passed`, `Failed`, `Error`, or `Timeout`. |
| `outputs` | Project-defined output values copied into the final node result. |
| `measurements` | One measurement object, an array of measurement objects, or `{ "items": [...] }`; PicoATE converts this into `MeasurementResult`. |
| `errorCode` | Stable machine-readable error or failure code. |
| `errorMessage` | Human-readable error or failure detail. |

Measurement object fields:

| Field | Meaning |
|-------|---------|
| `name` | Measurement/result name such as `VOUT` or `PackVoltage`. |
| `value` | Measured or decoded value. |
| `unit` | Display/reporting unit. |
| `rawValue` | Optional raw ADC/CAN/protocol value before scaling. |
| `lowerLimit` / `lower` / `min` | Optional lower limit. |
| `upperLimit` / `upper` / `max` | Optional upper limit. |
| `status` / `result` / `outcome` | `Passed`, `Failed`, `Error`, `Skipped`, or `Unknown`. |
| `passed` | Optional boolean shorthand; `false` maps to `Failed`. |
| `errorCode` / `errorMessage` | Optional measurement-specific diagnostic detail. |

When `status` is omitted, PicoATE derives a default status from the module
`outcome`. Extra fields are preserved in `MeasurementResult::attributes`.

Outcome semantics:

| Outcome | Meaning |
|---------|---------|
| `Passed` | Module action completed successfully. |
| `Failed` | Test limit or product condition failed; scheduler may apply fail policy. |
| `Error` | Module, instrument, protocol, or host error; scheduler may apply error policy. |
| `Timeout` | Module-level timeout; scheduler may apply timeout policy. |

## Minimal Python Example

See `examples/modules/echo_module.py`.

That script reads each request line, copies `context.inputs` to `outputs`, and
returns `Passed`. A project CAN module can follow the same shape while replacing
the echo logic with protocol parsing and device communication.

`examples/python_echo_sequence.json` invokes the Python module through
`QProcessTransport`:

```json
{
  "moduleBindings": [
    {
      "moduleId": "python.echo",
      "transport": "qprocess",
      "program": "${PYTHON_EXE}",
      "arguments": [
        "${PROJECT_DIR}/examples/modules/echo_module.py"
      ],
      "timeoutMs": 3000
    }
  ]
}
```

`PYTHON_EXE` can be provided by CMake during tests or by an environment/runtime
variable in a deployed project.

## DLL Bridge ABI

The first DLL bridge prototype uses a minimal C ABI:

```cpp
extern "C" __declspec(dllexport)
int PicoATE_Execute(
    const char* requestJsonUtf8,
    char* responseJsonUtf8,
    int responseBufferSize);
```

The request JSON is the same `ModuleTransportRequest` used by
`QProcessTransport`. The response buffer must be filled with one
`ModuleTransportResponse` JSON object encoded as UTF-8 and terminated with
`'\0'`.

Return code semantics:

| Return code | Meaning |
|-------------|---------|
| `0` | Function call succeeded; parse `responseJsonUtf8` for test outcome. |
| non-zero | Bridge-level DLL call failure; PicoATE reports `DllExecuteFailed`. |

Important limitation: the current `DllBridgeInvoker` runs in-process. It calls
the DLL on a worker thread and can report `DllExecuteTimeout`, but it cannot
safely terminate a stuck or crashed DLL. Production isolation should use a
future `PicoATE.NativeHost.exe` process that loads the DLL out-of-process.

`examples/nativehost_dll_sequence.json` shows the preferred DLL isolation path:

```json
{
  "moduleBindings": [
    {
      "moduleId": "native.dll.echo",
      "transport": "qprocess",
      "program": "${PICOATE_NATIVE_HOST}",
      "arguments": [
        "--dll",
        "${PICOATE_TEST_DLL}",
        "--symbol",
        "PicoATE_Execute"
      ],
      "timeoutMs": 3000
    }
  ]
}
```

In this mode, the main PicoATE process never loads the vendor DLL. If the DLL
hangs or crashes, the NativeHost process is the failure boundary.

## What Must Not Leak Into This Contract

- UI state or widgets.
- Scheduler internals such as `NodeActivation` or `NodeResult`.
- C++-only object layouts.
- Product-specific protocol assumptions in framework code.

PicoATE should be able to call any executable that follows this JSON contract,
regardless of whether it is written in C++, Python, C#, LabVIEW, or another
language.

## Built-In Language Support

PicoATE only provides built-in loading support for:

- C/C++ DLL modules through `PicoATE.NativeHost.exe`.
- Python scripts through `QProcessTransport` and a configured Python
  interpreter.

Other languages must be packaged by the project team as standalone `.exe`
programs that follow the stdin/stdout JSON contract. PicoATE does not provide
SDKs or templates for those languages.
