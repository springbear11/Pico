# PicoATE Module Runtime

The scheduler executes business logic through action modules. This keeps module
code independent from scheduler-owned runtime structures such as
`NodeResult`, `NodeActivation`, and `ExecutionSession`.

## Current Boundary

```text
ExecNode(Action)
  -> ActionNodeHandler
  -> ModuleRegistry
  -> IModule::execute(functionName, ModuleExecutionContext)
  -> ModuleResult
  -> NodeResult
```

`IModule` returns `ModuleResult`, not `NodeResult`. `ActionNodeHandler` owns the
translation into `NodeResult`, including `nodeId`, timestamps, and node outcome.

## Transport Boundary

Out-of-process or language-specific modules should be exposed to the scheduler
through `TransportModuleAdapter`. The adapter still implements `IModule`, so
`ActionNodeHandler` and `ModuleRegistry` do not need to know whether a module is
in-process, a DLL bridge, a Python host, or a child process.

```text
ActionNodeHandler
  -> ModuleRegistry
  -> TransportModuleAdapter
  -> IModuleTransport::call(ModuleTransportRequest, timeoutMs)
  -> ModuleTransportResponse
  -> ModuleResult
```

The transport DTOs are intentionally close to `ModuleResult`:

```cpp
struct ModuleTransportRequest {
    QString traceId;
    ModuleId moduleId;
    ModuleFunction functionName;
    ModuleExecutionContext context;
};

struct ModuleTransportResponse {
    ModuleOutcome outcome = ModuleOutcome::Passed;
    QVariantMap outputs;
    QVector<MeasurementResult> measurements;
    QString errorCode;
    QString errorMessage;
};

enum class ModuleTransportStatus {
    Ok,
    Timeout,
    TransportError
};
```

`traceId` is generated by `TransportModuleAdapter` for cross-process logging and
debugging. A successful transport call copies `ModuleTransportResponse` into
`ModuleResult`. A transport timeout maps to `ModuleOutcome::Timeout` with
`TransportTimeout`; a transport failure maps to `ModuleOutcome::Error` while
preserving the host-provided error code/message when present.

## QProcessTransport

`QProcessTransport` is the first concrete external-module transport. It starts a
short-lived child process for each module call, writes one JSON request line to
stdin, waits for one JSON response line from stdout, and maps process failures
to `ModuleTransportStatus`.

The JSON line protocol is documented in `docs/module_contract.md`. The built-in
`PicoATE.MockHost` executable and `examples/modules/echo_module.py` are minimal
reference implementations of that contract. `examples/python_echo_sequence.json`
demonstrates calling the Python example through `QProcessTransport`.

Sequences can declare external process bindings through top-level
`moduleBindings`. Runtime setup calls `registerConfiguredModules()` to resolve
placeholders, create `QProcessTransport`, wrap it in `TransportModuleAdapter`,
and register the adapter on `ExecutionSession`.

Placeholder replacement is centralized in `VariableResolver`, documented in
`docs/variable_resolver.md`. `ModuleBindingRegistrar` no longer owns a private
replacement implementation, so NativeHost manifests and future station/project
configuration can reuse the same rules and diagnostics.

## DllBridgeInvoker

`DllBridgeInvoker` is an in-process bridge for validating the DLL ABI before
moving DLL loading into `PicoATE.NativeHost.exe`. It loads a DLL with `QLibrary`,
resolves `PicoATE_Execute`, sends the same request JSON used by
`QProcessTransport`, and parses the returned response JSON.

The DLL call runs on a worker thread and is guarded by a timeout. If the timeout
expires, PicoATE returns `ModuleTransportStatus::Timeout` with
`DllExecuteTimeout`. The worker thread is only asked to interrupt; it cannot be
safely killed in-process. This is intentional and documents the remaining risk
that NativeHost must remove.

## NativeHost

`PicoATE.NativeHost.exe` is the out-of-process host for C/C++ DLL modules. It
uses the same stdin/stdout JSON protocol as `PicoATE.MockHost`, then loads the
configured DLL and calls the `PicoATE_Execute` ABI inside the child process.

```text
ActionNodeHandler
  -> TransportModuleAdapter
  -> QProcessTransport
  -> PicoATE.NativeHost.exe
  -> QLibrary
  -> VendorDriver.dll
```

This is the preferred path for production DLL integration because the main
PicoATE process does not load vendor DLLs into its own address space.

NativeHost can be configured either with legacy command-line `--dll` arguments
or with a manifest JSON file. The manifest path is preferred for project
delivery because DLL path, exported symbol, buffer size, timeout, and metadata
stay in a reusable configuration file while sequence JSON only references that
file. See `docs/nativehost_manifest.md`.

## Simulated CAN DLL Example

`examples/can_dll_sequence.json` demonstrates the intended project delivery
shape without requiring a CAN analyzer or real hardware:

```text
Sequence JSON
  -> moduleBindings
  -> QProcessTransport
  -> PicoATE.NativeHost.exe
  -> examples/nativehost/can_decode_manifest.json
  -> PicoATE.CanExampleModule.dll
```

The DLL decodes a configured CAN frame from software inputs, applies scale and
offset, evaluates min/max limits, and returns both `outputs` and
`measurements`. This verifies the framework boundary first. A real project can
later replace only the DLL and CAN parser logic while keeping the UI, scheduler,
sequence JSON shape, and NativeHost path stable.

## ModuleResult

```cpp
enum class ModuleOutcome {
    Passed,
    Failed,
    Error,
    Timeout
};

struct ModuleResult {
    ModuleOutcome outcome = ModuleOutcome::Passed;
    QVariantMap outputs;
    QVector<MeasurementResult> measurements;
    QString errorCode;
    QString errorMessage;
};
```

`Failed`, `Error`, and `Timeout` intentionally remain distinct because error
policy decisions depend on the difference between a test limit failure, a module
or instrument error, and a timeout.

## MeasurementResult

Measurements are represented inside the scheduler as `MeasurementResult`
values, not ad-hoc maps:

```cpp
struct MeasurementResult {
    QString name;
    QVariant value;
    QString unit;
    QVariant rawValue;
    bool hasLowerLimit = false;
    double lowerLimit = 0.0;
    bool hasUpperLimit = false;
    double upperLimit = 0.0;
    MeasurementStatus status = MeasurementStatus::Unknown;
    QString errorCode;
    QString errorMessage;
    QVariantMap attributes;
};
```

External modules may still return the compact JSON object form documented in
`docs/module_contract.md`; PicoATE converts it to `MeasurementResult` when the
response is parsed. `NodeResult`, `AttemptReport`, and `StepReport` all expose
structured measurement vectors so CLI, reports, and future UI ViewModels do not
need to read `outputs["measurements"]`.

## JSON Action Fields

Action steps can select a module explicitly:

```json
{
  "id": "measure-voltage",
  "kind": "action",
  "moduleId": "mock.measurement",
  "function": "measureVoltage",
  "inputs": {
    "outputs": {
      "actualVoltage": 4.999
    },
    "measurements": {
      "name": "VOUT",
      "value": 4.999,
      "unit": "V"
    }
  }
}
```

If `moduleId` is omitted, `ActionNodeHandler` uses the built-in `mock.action`
module for backward-compatible examples and tests.

Cleanup steps normally behave as no-op cleanup markers. If a cleanup node has a
`moduleId`, `ActionNodeHandler` executes it through the same module path as an
action step. This allows real shutdown work such as instrument `close`, fixture
release, or power-off commands to live in cleanup while preserving old no-op
cleanup examples.

Before a node handler runs, `NodeRunner` resolves runtime variables on a
temporary copy of the node payload. This allows action inputs such as
`${var.sampleIndex}`, `${loop.value}`, `${uut.id}`, and `${attempt.number}` to
drive module calls without mutating the immutable `ExecutionPlan`. Whole-field
variables preserve their original type; embedded variables produce strings.

## Built-In Mock Modules

The default `NodeRunner` registers:

| Module id | Purpose |
|-----------|---------|
| `mock.action` | Backward-compatible generic mock action. |
| `mock.measurement` | Explicit mock measurement action for JSON examples. |
| `example.dmm` | Built-in DMM adapter spike that uses runtime device services. |
| `example.can` | Built-in CAN adapter spike that uses runtime device services. |

Both built-in mock modules support the same control fields through either
top-level legacy parameters or `inputs`:

| Field | Behavior |
|-------|----------|
| `outcome` | `Passed`, `Failed`, `Error`, or `Timeout`. |
| `failForUut` | Fails only the matching UUT id. |
| `failUntilAttempt` | Fails attempts up to the given zero-based attempt index. |
| `outputs` | Copied to `ModuleResult::outputs`. |
| `measurements` | Parsed into `QVector<MeasurementResult>`. |
| `errorCode` / `errorMessage` | Copied to the final `NodeResult`. |

`ActionNodeHandler` also mirrors measurements into
`NodeResult::outputs["measurements"]` for backward compatibility with early
examples. New consumers should read `NodeResult::measurements` or
`ExecutionReport` measurement fields.

## 业务模块访问逻辑设备

现在 `ModuleExecutionContext` 会携带 `IModuleRuntimeServices* runtimeServices`。
业务模块不需要自己创建 DMM/CAN/电源 driver，也不需要知道 station config 的解析细节。

推荐路径是：

```text
Action step
  -> example.dmm / project.xxx module
  -> context.runtimeServices
  -> DeviceSessionManager
  -> TransportDeviceSession
  -> PersistentQProcessTransport
  -> Instrument Host
```

关键接口：

```cpp
class IModuleRuntimeServices {
public:
    virtual DeviceSessionOpenResult openDeviceSession(const DeviceId& deviceId) = 0;
    virtual DeviceSessionError closeDeviceSession(const DeviceId& deviceId) = 0;
    virtual std::shared_ptr<IDeviceSession> deviceSession(const DeviceId& deviceId) const = 0;
    virtual ModuleResult invokeDevice(const DeviceId& deviceId,
                                      const ModuleFunction& functionName,
                                      const QVariantMap& inputs,
                                      const ModuleExecutionContext& context) = 0;
};
```

`TransportDeviceSession` 是当前的 fake/host-backed device session。它把
`DeviceSessionManager` 的 `open/close/health` 和业务命令转成
`IModuleTransport` 请求。这样 station config 中的 `DMM1 -> fake.dmm` 可以落到
一个长驻 fake host；未来也可以替换成真实 DMM/CAN/电源 host。

当前内置 adapter 只用于 spike，不是最终项目 SDK：

| Module id | 已验证函数 | 说明 |
|-----------|------------|------|
| `example.dmm` | `ConnectDMM` / `Disconnect` / `ConfigureDCV` / `ReadDMM` / `GetIdentity` / `MeasureDCV` | 通过逻辑设备 `DMM1` 调 fake DMM host。 |
| `example.can` | `ConnectCAN` / `Disconnect` / `ReadFrame` | 通过逻辑设备 `CAN1` 调 fake CAN host。 |

示例链路：

```text
examples/dmm_can_adapter_sequence.json
  + examples/stations/basic_station.json
  -> example.dmm / example.can
  -> PicoATE.FakeInstrumentHost.exe
```

这证明了 DMM/CAN adapter 的形状是可落地的，但仍然不是“真实硬件已经验证”。
真实项目接入时，应把 `example.dmm/example.can` 替换成项目自己的适配模块或
真实长驻 Instrument Host。

## Future NativeHost Path

Future DLL/Python/LabVIEW isolation can reuse the same boundary by implementing
one or more `IModuleTransport` backends:

| Transport | Purpose |
|-----------|---------|
| Fake transport | Unit tests for adapter mapping. |
| QProcessTransport | Implemented JSON over stdio transport to a standalone host executable. |
| PersistentQProcessTransport | Implemented long-lived JSON over stdio transport for host processes that keep session state across calls. |
| DllBridgeInvoker | Implemented direct `QLibrary` load for early DLL ABI validation without isolation. |
| NativeHost executable | Implemented child process host that loads DLLs out-of-process. |
| PythonHostTransport | Child process host for Python-based modules. |

Each backend should return `ModuleTransportStatus` plus
`ModuleTransportResponse`; the adapter remains the single place that translates
transport behavior back into scheduler-neutral `ModuleResult`.

## NativeHost DLL Session Lifetime

需要跨 Step 保持厂商句柄的 DLL 必须使用 `persistent-qprocess`。`DllBridgeInvoker`
对 NativeHost 内加载的 DLL 设置 `QLibrary::PreventUnloadHint`，因此 DLL 静态状态和
设备句柄会保持到 Host 进程退出：

```text
Setup/open -> Main/read/write -> Cleanup/close
```

普通 `qprocess` 每次调用都会创建新 Host，不能用于依赖跨 Step 句柄的硬件适配器。
模板见 `templates/can_hardware_dll/`。
